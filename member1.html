Drone Delivery Routing

TSP Tour
The Business Case

An e-commerce provider wants to deploy a fleet of delivery drones to service last-mile packages in a dense urban neighborhood. Drones have limited battery life and payload constraints, and need to return to recharge between sorties. The business goal is to minimize total flight time and battery consumption while meeting delivery time windows and avoiding restricted airspace, so operators can complete more deliveries per drone per day and reduce operational costs.

Graph Modeling

Nodes: Depot(s) (launch/charge points) and customer delivery locations. Edges: Feasible flight segments between nodes. Weights: Estimated flight time, energy cost, or distance for each segment (considering wind, altitude changes, and no-fly detours). Constraints: Per-route battery capacity (max path length), payload limits, time windows, and forbidden zones (edges removed or given high cost).

Algorithm Analysis

Model is a constrained Traveling Salesman Problem (TSP). For small routes, exact solvers (MILP / branch-and-bound) produce optimal tours; for practical scale use fast heuristics: nearest neighbor, 2-opt/3-opt, or metaheuristics (genetic, simulated annealing) and specialized vehicle-routing solvers that respect battery/time windows. Combine with dynamic replanning to react to wind or temporary no-fly zones. Use route clustering (split region into sub-tours) to scale across many deliveries.

Implementation Logic
> > INIT all delivery points as nodes > CALCULATE distance between every pair of points > SET Base = 0 > MARK all nodes as unvisited > CURRENT = Base > WHILE unvisited nodes remain: v = Nearest_Unvisited(CURRENT) ADD v to Route MARK v as visited CURRENT = v > RETURN to Base > OUTPUT Final_Tour

Metro City Navigator

Dijkstra’s Algo
The Business Case

A transit app wants to give commuters the fastest door-to-door route during peak hours using real-time train frequencies, walking times, and transfer penalties. The aim is to reduce passenger travel time and improve service satisfaction by routing riders through optimal platforms and transfer sequences, even when disruptions occur.

Graph Modeling

Nodes: Stations and platform access points (optionally separate nodes for entrances/exits). Edges: Track segments, inter-station links, and pedestrian transfer corridors. Weights: Travel time along the edge (train run time + waiting time estimate + transfer penalty). Real-time adjustments made from live schedules and crowding data. Constraints: Line closures or service suspensions implemented by disabling edges.

Algorithm Analysis

Dijkstra’s single-source shortest path is ideal for finding the fastest route from a passenger’s origin to destination in a weighted graph with nonnegative travel times. For real-time routing, run Dijkstra on an up-to-date graph each query. For many simultaneous queries, use A* with a geographic heuristic to speed up lookup, or precompute frequent origin/destination pairs. Handle time-dependent edge weights (timetables) with time-expanded graphs or time-dependent Dijkstra variants.

Implementation Logic
> > INIT all distances = Infinity, Source = Start_Station > PUSH Source to Min-Priority Queue > WHILE Queue not empty: u = Extract Min FOR v in neighbors(u): IF Time[u] + TravelTime < Time[v]: Time[v] = New_Time Parent[v] = u > BACKTRACK from Destination to Source > OUTPUT Shortest_Metro_Route

International Trade Hub

Floyd–Warshall
The Business Case

A global logistics operator wants an internal service that can quickly evaluate the best multi-stop shipping paths between any pair of ports, taking into account transit time, transshipment costs, and port handling delays. The product supports contract negotiation, route contingency planning, and fast rerouting during disruptions.

Graph Modeling

Nodes: Ports, major transshipment hubs, and regional distribution centers. Edges: Direct shipping legs between nodes. Weights: Composite cost per leg (time + cost + reliability penalty). Goal: Be able to answer “best connection between any two ports” queries instantly, including the best intermediate hubs.

Algorithm Analysis

Floyd–Warshall computes all-pairs shortest paths and is convenient if the graph is moderate in size and you need constant-time lookup between any two ports. It also produces intermediate-node information for path reconstruction. For very large graphs, use repeated Dijkstra runs from key hubs or a hierarchical approach (contracted graphs). Extend the model to support multi-criteria optimization (time vs cost) by running separate all-pairs computations or using multi-objective techniques.

Implementation Logic
> > INIT Dist[i][j] for all ports > FOR k in ports: FOR i in ports: FOR j in ports: IF Dist[i][k] + Dist[k][j] < Dist[i][j]: Dist[i][j] = Updated_Distance > OUTPUT All_Pairs_Shortest_Paths

Smart Forest Monitoring

Kruskal’s MST
The Business Case

A conservation authority needs to deploy sensor towers across a large forest to detect illegal logging and monitor wildlife. Cabling (or low-power mesh backhaul) is expensive to install. The business aim is a connectivity plan that links all sensors with minimum installation cost so monitoring is affordable and maintainable.

Graph Modeling

Nodes: Candidate sensor tower sites (clearings, existing structures). Edges: Potential direct links (cable runs or line-of-sight radio links) between sites. Weights: Installation cost (distance × terrain multiplier), maintenance risk, or signal degradation. Constraints: Avoid environmentally sensitive corridors, respect terrain obstacles (infinite cost/disallowed edges).

Algorithm Analysis

Kruskal’s Minimum Spanning Tree builds the lowest-cost network that connects all sensor sites. It iterates edges in increasing weight and uses a Union-Find structure to avoid cycles — ideal when edge list is sparse or when you can enumerate candidate links. If multiple constraints exist (bandwidth or redundancy), augment MST with extra links for resilience or compute k-edge connected subgraphs. Kruskal is simple to implement and yields a clear procurement bill of materials.

Implementation Logic
> > INIT each tower as its own set > SORT all edges by installation cost > FOR each edge (u, v) in sorted_edges: IF Find(u) != Find(v): ADD edge to MST Union(u, v) > OUTPUT Minimum_Cost_Sensor_Network

Renewable Energy Grid Planner

Prim’s MST + TSP
The Business Case

A renewable operator wants to connect a field of wind turbines (or solar arrays) to a substation at minimal cable cost and then schedule periodic maintenance inspections that visit every turbine and return to base. The project target is minimized infrastructure cost plus an efficient inspection route that cuts crew travel time.

Graph Modeling

Nodes: Turbine locations and the main substation. Edges: Possible cable routes between turbines and between turbines and the substation (considering terrain and right-of-way). Weights: Cost to lay cable (distance × terrain factor) for MST; for inspection routing, use travel time between turbines as edge weights for the TSP. Constraints: Redundancy requirements (optional second path), maintenance time windows, and access road availability.

Algorithm Analysis

First solve an MST (Prim’s algorithm recommended when you have a dense, location-based graph and want a single growing tree from the substation) to produce a minimum-cost cable layout. Prim runs efficiently with a priority queue and is straightforward to adapt to geo-constraints. For crew inspection cycles, compute a near-optimal TSP tour over turbine nodes using MST-based heuristics (e.g., perform a preorder traversal of the MST to get a short tour, then improve with 2-opt/3-opt), or feed MST result to Christofides/heuristics for better guarantees. If inspection teams operate in parallel, partition into multiple tours (multi-TSP / vehicle routing).

Implementation Logic
> > INIT all nodes, Start = Main_Power_Node > MARK Start as connected > WHILE unconnected nodes remain: SELECT cheapest edge from Connected → Unconnected ADD edge to MST MARK new node as connected > PERFORM simple DFS on MST to form Maintenance_Tour > OUTPUT MST + Tour_Path
